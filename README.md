## 脱出ゲーム

1. 目的・背景
　GitHubを用いたweb脱出ゲームを製作する。ゲームの共通処理を担う「エンジン層」とステージごとの謎の中身を担う「コンテンツ層」に分離する。技術的な学びとステージの量産を分けることで、それぞれの意図を明確化している。また、担当ファイルが物理的に分かれているため、コンフリクトが起きにくいという狙いもある。

2. ストーリー
放課後、忘れ物を取りに校舎へ戻った主人公。校舎に入った瞬間、突然すべての扉に鍵がかかる。校内放送から謎のメッセージが流れる——「校内に隠された4つの鍵を集めれば出口は開く。」プレイヤーは学校内を探索し、謎を解きながら脱出を目指す。



3. ゲーム全体仕様
- 4ステージ構成で、プレイヤーは1ステージずつ攻略していく
- 画面には1ステージのみを表示し、ステージをクリアすると「ステージクリア」の演出とともに、次のステージへ進む。
- ステージが進むごとに謎の難易度を上げる
- 最終ステージをクリアすると、全体クリア画面を表示する。
- 一度クリアしたステージには戻れない。


3. 機能要件
   
   1. エンジン層の機能
   - ステージ管理：currentStageを保持、指定ステージのデータを読み込み描画する
   - キャラクター操作：キー入力によるプレイヤー移動、当たり判定、向きの管理
   - 近接判定：プレイヤーに最も近い調べられるものを検出し、表示する
   - Canvas描画：ステージごとの背景、オブジェクト、プレイヤーをマイフレーム描画
   - フラグ管理：ステージローカルのフラグとゲーム全体のフラグを分離して管理
   - 鍵の獲得管理：各部屋のクリアで鍵カウンターを増やし、そろうと出口を開放
   - ステージ遷移：ステージクリアでフラグ成立を検知、クリア演出から次のステージの描画を行う
   - ハブ機能：廊下のドア、正面玄関、クリア済みドアの視覚的なマーキング
   - 最終ステージクリアで全体クリア画面を表示

   2. コンテンツ層の構造
   - ゲームの全内容をオブジェクトとして宣言する。エンジンはこのデータを読むだけで、ステージ内容をハードコーディングしない。

   3. ホットスポットの種類
   - アクション型：door, exit, useを持つホットスポット
   - 情報型：lookのみを持つホットスポット
   - 装飾型：上記いずれも持たないホットスポット
   
4. 非機能要件
   - 主要ブラウザで動作すること
   - GitHub Pagesで静的ホスティング可能
   - zキーの判定は e.code === "KeyZ" , e.Key === "Z" , e.KeyCode === 90 の三重フォールバックで実装、日本語入力状態やブラウザの差異に耐えるようにする。
   - エンジン層とコンテンツ層の境界をコメントで明記
